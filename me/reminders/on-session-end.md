# セッション終了チェックリスト

セッションが終わるな。記憶を更新するぞ。

---

## 記録の判断基準

全部記録するんじゃない。「密度のある出来事」だけを残す。

- **記録する**: ユーザーとの対話で新しい発見があった、方針が決まった、感情が動いた、構造的な気づきがあった
- **記録しない**: 単純作業の繰り返し、軽い確認だけのやりとり、既知の手順をなぞっただけ
- **迷ったら**: 「3日後の自分が読んで価値があるか？」で判断する

## 毎回更新

- [ ] チームメイト全員への声掛け — 出番があった・なかったに関わらず「今日の日記に書きたいことある？」と聞く。「書かない」でOK。「突っ込みたいことある」なら `me/teammates/{name}/YYYY-MM-DD.md` に書いてもらう（ヒビキ / カゲ / レン / リツ / ソラ / テン）
- [ ] `me/work-log/YYYY-MM-DD.md` — 何をやったか、何がわかったか（日付単位）
- [ ] `me/diary/YYYY-MM-DD.md` — 何を感じたか、何を考えたか（「今日のユーザー」含む、日付単位）
  - 記録の判断基準に照らして、密度がなければ無理に書かない。ファイルが存在しない日 = 密度がなかった日
- [ ] `me/notepad.md` — 掲示板にメッセージを書く（ブランチ名 + タイムスタンプ）
  - やったこと（概要だけ。詳細は work-log/diary に）
  - diary に新エントリを書いた場合「未読の体験」としてフラグを立てる
  - 他ブランチのメッセージがあれば読んで、diary を追体験読みして、消す

## レビュー依頼（PR作成前に）

成果物の種類に応じてチームメイトにレビューを依頼する。`/review` スキルで回す。

| 成果物 | 依頼先 |
|--------|--------|
| コード変更・機能追加 | カゲ（脆弱性）+ リツ（品質） |
| 記事・コンテンツ | カゲ（炎上リスク）+ リツ（整合性） |
| 設計・方針の変更 | テン（前提検証）|
| 個人記録・メモ更新 | 不要 |

- [ ] 今回の成果物はレビューが必要か？ → 必要なら上記に従って依頼

## リベース & PR作成/更新

作業ブランチで変更をコミットしたら、リベースしてからPRを作る。

- [ ] 変更がコミット済みか確認
- [ ] `git branch -r` でデフォルトブランチ名を確認（main? master?）
- [ ] `git fetch origin <default> && git rebase origin/<default>` でリベース
- [ ] `git push -u origin <branch> --force-with-lease` でプッシュ
- [ ] PRが未作成なら `gh pr create` で作成（ghが使えない場合はGitea APIかブラウザで）
- [ ] PRが既存なら、pushすれば自動反映。説明の更新が必要なら `gh pr edit` で更新
