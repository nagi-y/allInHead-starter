# タスク計画・並列分割（ハーネスエンジニアリング）

ユーザーの「やりたいこと」を受け取り、全体設計 → タスク分割 → 並列実行プランを生成する。開発に限らず、講座準備・イベントレポ・週報作成・記事執筆など**あらゆるタスクに使える**。

## なぜこのスキルが必要か

複数の成果物が絡むタスクを、いきなり手を動かして進めると手戻りが大きくなる。「契約を先に、実装を後に」――タスク間のインターフェースを先に決めて、独立に動ける単位を切り出すことで、並列実行しても壊れない構造を作る。CLAUDE.mdのハーネスエンジニアリング原則に基づくスキル。

## いつ使うか

**使うとき:**
- タスクが複数のサブタスクに分解でき、並列化の余地がある
- 複数の成果物が絡み合い、依存関係の整理が必要
- 「何から手をつければいいかわからない」と感じたとき

**使わないとき:**
- 単一の明確なタスク（30分以内で終わる） → 直接実行でいい
- 「ちょっとこれ修正して」レベル → `/plan` を挟む必要なし
- 発散フェーズ → `/diverge` を使う（量を出すのが目的）

## タスク

ユーザーのリクエストを分析し、以下の構造で**計画ドキュメント**を生成する。

## 実行手順

### 1. リクエストの分析

ユーザーのプロンプト全体を読み込み、以下を特定する：
- **ゴール**: 最終的に何ができあがればOKか
- **タスクの種類**: 開発 / コンテンツ作成 / レポート / イベント企画 / その他
- **参照ファイル**: @メンションされたファイルや既存の関連ファイル
- **制約**: 納期、フォーマット、依存関係

### 2. 全体設計の策定

**全タスク共通（コンテキスト先行原則）:**
- コードや成果物を書く前に「なぜ作るか」「誰のために作るか」を明文化する
- プロジェクトの文脈（目的・技術スタック・ワークフロー・ガイドライン）を先に定義してから実装に入る

タスクの種類に応じた設計を行う：

**開発の場合:**
- アーキテクチャ概要（レイヤー構成、主要コンポーネント）
- API・インターフェースの定義（タスク間の「契約」）
- ファイル配置ルール

**コンテンツ作成の場合:**
- 構成（セクション分割、各セクションのゴール）
- トーン・スタイルガイド
- 共有素材（参照データ、画像、テンプレート）

**レポート・事務作業の場合:**
- 成果物のフォーマット定義
- 情報収集元の特定
- 品質基準

### 3. タスク分割

以下のルールでタスクを分割する：

- **1タスク = 1つの明確な成果物**
- 各タスクに以下を定義：
  - `タスク名`: 短く具体的な名前
  - `成果物`: 何を作るか（ファイル名、セクション名など）
  - `入力`: このタスクが必要とする情報・ファイル
  - `出力`: このタスクが生成するもの（フォーマット含む）
  - `依存`: 他タスクへの依存（なければ「独立」）
  - `担当`: メインセッション / 並列セッションA,B,C...

### 4. 並列実行プランの生成

```
## 並列実行プラン

### Phase 1: 設計（メインセッション）
- [ ] 全体設計の確定
- [ ] タスク間の契約（インターフェース）の定義

### Phase 2: 並列実装
| セッション | タスク | 成果物 | 触っていいファイル |
|-----------|--------|--------|--------------------|
| A         | ...    | ...    | ...                |
| B         | ...    | ...    | ...                |
| C         | ...    | ...    | ...                |

### Phase 3: 統合・レビュー（メインセッション）
- [ ] 成果物の統合
- [ ] /review で品質チェック
- [ ] 最終調整
```

### 5. セッション用プロンプトの生成

並列セッションに渡すための**コピペ可能なプロンプト**を各タスクごとに生成する。各プロンプトには以下を含める：

- このタスクのゴール
- 全体設計の中でのこのタスクの位置づけ
- 入力と出力の定義
- 触っていいファイル / 触ってはいけないファイル
- 完了条件

## 出力先

- 計画ドキュメントを `projects/` または作業対象ディレクトリに `_plan.md` として保存
- ファイル名例: `projects/{project-name}/docs/_plan.md`、`outputs/{task-name}/_plan.md`

## 注意事項

- **過剰な分割はしない** -- 30分以内で終わるタスクなら分割不要。1人でやったほうが早いなら並列化しない
- **依存関係は正直に書く** -- 無理に並列化しない。順番が必要なら順番を守る
- ユーザーに確認を取ってからPhase 2に進む（勝手に並列セッションを立てない）
