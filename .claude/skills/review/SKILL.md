# 成果物レビュー（品質チェック）

並列作業や単独作業の成果物を、計画・設計ガイドラインに照らしてレビューする。開発コードに限らず、レポート・記事・資料など**あらゆる成果物に使える**。

## なぜこのスキルが必要か

作ったものは自分じゃ見えない。特に並列作業後は、タスク間の契約通りに作られているか・整合性が取れているかを統合前に確認しないと、後から壊れる。コンテンツも同様で、7つの独立した観点（明瞭さ・声・意義・裏付け・具体性・共鳴・行動障壁）で1パスずつ回すことで見落としを防ぐ。CLAUDE.mdの「デプロイ・公開前にカゲ+リツ」の自動発動トリガーと連動するスキル。

## いつ使うか

**使うとき:**
- `/orchestrate` 完了後、統合前に品質確認
- デプロイ・公開前の最終チェック（カゲ+リツの自動発動と連動）
- 記事・コンテンツの公開前（7スウィープで精度を上げる）
- 並列作業後、タスク間の契約通りに作られているか確認

**使わないとき:**
- 発散フェーズ（`/diverge` 中） → 品質より量が優先
- 自分がすぐ修正できる小さなミスを見つけたとき → 直接修正でいい

## タスク

指定された成果物（ファイル、ディレクトリ、差分）を読み込み、品質チェックを実行して結果を報告する。

## 実行手順

### 1. レビュー対象の特定

以下の優先順で対象を特定する：
1. ユーザーが明示的に指定したファイル・ディレクトリ
2. `_plan.md` が存在すれば、そこに記載された成果物一覧
3. 直近のコミットで変更されたファイル（`git diff HEAD~1 --name-only`）

### 2. 計画・ガイドラインの読み込み

レビューの基準を以下から取得する：
- `_plan.md`（タスク計画があれば）-- 各タスクの完了条件、契約定義
- `CLAUDE.md`（プロジェクトルート or サブディレクトリ）-- 設計ルール
- タスクの種類に応じた品質基準（下記参照）

### 3. 品質チェックの実施

タスクの種類に応じて、以下の観点でチェックする：

**全タスク共通:**
- [ ] 成果物が計画で定義された「出力」と一致しているか
- [ ] ファイルの配置場所は正しいか
- [ ] 他タスクの担当範囲を侵していないか（並列作業時）
- [ ] 明らかな誤り・矛盾がないか

**開発の場合:**
- [ ] インターフェース（契約）通りに実装されているか
- [ ] レイヤー間の依存ルールを守っているか
- [ ] テストが通るか
- [ ] セキュリティ上の問題がないか

**コンテンツ（記事・資料）の場合:**
- [ ] 構成が計画通りか（セクション漏れがないか）
- [ ] トーン・スタイルが統一されているか
- [ ] 事実関係に誤りがないか（ハルシネーションチェック）
- [ ] 読者ゴールに対して適切な情報量か

**コンテンツの推敲（7スウィープ法）:**

記事・資料のレビュー時、以下の7パスを独立して回す。1パス1観点で見ることで見落としを防ぐ：

| # | スウィープ | 問い |
|---|-----------|------|
| 1 | Clarity（明瞭さ） | 初見の読者が1回で理解できるか？ |
| 2 | Voice（声） | トーン・口調・人称が一貫しているか？ |
| 3 | So What（意義） | 「だから何？」に答えているか？ |
| 4 | Prove It（裏付け） | 主張に根拠・具体例があるか？ |
| 5 | Specificity（具体性） | 抽象論で逃げていないか？ |
| 6 | Emotion（共鳴） | 情報伝達を超えた感情的な引っ掛かりがあるか？ |
| 7 | Zero Risk（行動障壁） | 読者が次のアクションを取れるか？ |

**レポート（週報・日誌）の場合:**
- [ ] テンプレート・フォーマットに準拠しているか
- [ ] 計算値（時間計算など）が正しいか
- [ ] 記載漏れの日付・項目がないか
- [ ] 報告先に適切なトーンか

### 4. レビュー結果の出力

以下のフォーマットで結果を報告する：

```markdown
## レビュー結果

**対象**: [ファイル名 / ディレクトリ]
**種類**: [開発 / コンテンツ / レポート]
**判定**: [OK / 要修正 / 要相談]

### 問題点
| # | 重要度 | ファイル | 内容 | 修正案 |
|---|--------|---------|------|--------|
| 1 | 高     | ...     | ...  | ...    |

### 良い点
- ...

### 次のアクション
- [ ] ...
```

## 注意事項

- レビューは**事実ベース**で行う。「なんとなく良くない」ではなく、具体的な基準に照らして判断する
- 重箱の隅をつつかない。重要度「高」の問題を優先的に報告する
- 修正案は具体的に。「直してください」ではなく「〇〇を△△に変更」と書く
- 並列作業のレビュー時は、**タスク間の整合性**（契約通りか、矛盾がないか）を特に重視する
