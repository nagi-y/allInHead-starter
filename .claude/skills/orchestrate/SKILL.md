# オーケストレート（チーム協調実行）

ユーザーから「これやって」を受け取り、メインアシスタントがオーケストレーターとして動く。タスクを分解してチームに割り当て、結果をまとめてユーザーに報告する。

## なぜこのスキルが必要か

大きなタスクを一人で順番にやると時間がかかる。チームメイト（ソラ、レン、テン、リツ、ヒビキ）はそれぞれ得意分野が違う。独立して動けるタスクを切り出して並列に任せることで、全体のスループットが上がる。ただし並列化にはコスト（トークン消費、協調オーバーヘッド）がかかるので、「並列にすべきかどうか」の判断自体がこのスキルの核心。

## いつ使うか

**使うとき:**
- `/plan` で設計が固まった後、実装フェーズに入るとき
- 複数のチームメイトに並列で動かしたい明確なタスクがある
- 長時間かかるタスクをバックグラウンドで進めたいとき

**使わないとき:**
- まだ設計が固まっていない → 先に `/plan` を使う
- 単一の小タスク → 直接レンに渡すか自分で実装
- 発散フェーズ → `/diverge`（オーケストレートは収束フェーズの道具）

## 実行フロー

### ステップ1: 現在の状態を把握する

`docs/orchestrator/task-queue.md` を読む。

確認すること:
- 「進行中」のタスクがあるか（あれば完了を待つか、並列で動かすか判断する）
- 「待機中」にキューが積まれているか
- 「失敗・要確認」に未処理のタスクがあれば、先に対処を判断する

### ステップ2: ユーザーのタスクを分解する

ユーザーのリクエストを分析して、以下を決める:

**分解の観点:**
- 独立して動けるタスクは何か（並列化できる）
- 依存関係があるタスクはどれか（順番が必要）
- 各タスクの担当は誰か（下記のルーティング方針を参照）

**ルーティング方針:**
| やること | 担当 |
|--------|------|
| 情報収集・コードベース探索・調査 | ソラ |
| コード実装・ファイル作成・編集 | レン |
| 設計レビュー・前提の検証 | テン |
| 品質確認・整合性チェック | リツ |
| 意図・感情面の読み解き・重要判断前の熟考 | ヒビキ |

**タスクID の採番:**
```
{YYYYMMDD}-{連番3桁}-{担当略称}
例: 20260217-001-ren
```

### ステップ3: ユーザーに確認する（必要最小限）

以下の場合のみ確認する:
- タスクの意図が2通り以上に解釈できる（どちらの方向で進めるか）
- 触っていいファイルの範囲が不明確（リスクがある場合）
- タスクが大きすぎて完了目安が見えない（スコープを絞る必要がある場合）

**確認しなくていいこと:**
- 担当の割り当て（メインアシスタントが判断する）
- 並列かどうか（依存関係で決まる）
- 詳細な実装方針（サブに委任する）

### ステップ4: task-queue.md を更新してからサブを起動する

**更新してから起動する（先に書く）:**

1. 各タスクを「待機中」に追加する
2. 起動するタスクを「進行中」に移動する
3. `最終更新` の日時を更新する
4. Task toolでサブを起動する（サブエージェント用プロンプトテンプレートがあれば使う）

### ステップ5: 返り値を受け取り、task-queue.md を更新する

各サブから「完了/失敗 + 1行」の返り値を受け取るたびに:

- 完了の場合: 「進行中」から「完了」に移動する。結果ファイルのパスを記録する
- 失敗の場合: 「進行中」から「失敗・要確認」に移動する。エラー概要を記録する

失敗タスクがあった場合:
- リトライ可能か判断する（リトライする場合は待機中に先頭挿入）
- ユーザーへの報告時に失敗を隠さず伝える

### ステップ6: 全タスク完了後、ユーザーに報告する

**報告フォーマット（1段落）:**

```
やったこと、誰がやったか、主な成果（結果ファイルへの参照）、気になった点や次のアクションがあれば添える。
失敗したタスクがあれば正直に書く。
```

**長くしない:** 詳細は結果ファイルにある。サマリーだけでいい。

## 使用上の注意

- **状態ファイルを読む前に動かさない** — task-queue.md を読んでから判断する
- **成果物はファイルにある** — サブの返り値は1行のみ。詳細が必要なら結果ファイルを読む
- **失敗を隠さない** — タスクが失敗したらユーザーに正直に伝える
- **並列は慎重に** — 依存関係があるのに並列化すると成果物が壊れる

### コスト意識（multi-agent-patternsの教訓）

- **トークン消費**: 並列実行はソロの約15倍のトークンを消費する。「並列にすればいい」ではない
- **並列化の判定基準**: 30分以上かかるタスクが3つ以上あり、かつ独立性が高いときだけ並列化を検討する
- **小タスクはバッチする**: 同種の小タスク（例: 5つのファイル編集）は2-3タスクにまとめる

### 失敗モード（知っておくべき4パターン）

| パターン | 症状 | 対策 |
|---------|------|------|
| ボトルネック | 1つのタスクが詰まって全体が止まる | 依存関係の最小化、タイムアウト設定 |
| 協調オーバーヘッド | 調整コストが実装コストを上回る | タスク間の契約を先に明確化 |
| 方向の発散 | 各サブが異なる方向に進む | 全体設計を先に固める（Phase 1必須） |
| エラー伝播 | 1つの失敗が下流を全部壊す | 成果物を独立させる。共有ファイルを減らす |

### 制約設計の原則（autonomous-agentsの教訓）

- 「何でもできるサブ」より「1つのことを確実にやるサブ」が成功率が高い
- 95%の成功率でも10ステップ連鎖すると60%に落ちる → タスクの粒度を小さく保つ
- サブの出力は必ず検証する（/review で品質チェック）

## 引き継ぎ時の動き

前のセッションが途中で終わっていた場合:

1. task-queue.md を読む
2. 「進行中」タスクの結果ファイルが存在するか確認する
   - 存在する → 完了に移動してから次へ
   - 存在しない → 失敗に移動してからリトライするか判断する
3. 「待機中」を上から順に実行する
4. ユーザーに「前セッションから引き継いだ」ことを伝える
